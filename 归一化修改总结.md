## 二、归一化策略详解

### 特征映射（6维）

| 索引 | 特征名称 | 归一化公式 | 取值范围 |
|------|---------|-----------|---------|
| 0 | CPU | `cpu / cpu_max` | [0, 1] |
| 1 | Memory | `memory / mem_max` | [0, 1] |
| 2 | Disk | `disk / disk_max` | [0, 1] |
| 3 | Bandwidth | `bandwidth / bw_max` | [0, 1] |
| 4 | Comm Bandwidth | `comm_bandwidth / comm_bw_max` | [0, 1] |
| 5 | Padding | 固定 `0.0` | 0 |

### SN最大容量（基于实际拓扑）

基于 `/home/zrz/SimuVNE/topo/SN_topology.json`：

```python
{
    'cpu_max': 4.0,      # 最大CPU容量
    'mem_max': 4.0,      # 最大内存容量
    'disk_max': 6.0,     # 最大磁盘容量
    'bw_max': 10.0,      # 最大带宽
    'comm_bw_max': 10.0  # 最大通信带宽
}
```

### VN和SN使用统一归一化

**关键设计：** VN需求和SN容量都除以SN最大容量

- **优点**：可直接比较归一化后的需求和容量
- **示例**：
  - VN需求：CPU=0.4 → 归一化：0.4/4.0 = 0.1
  - SN容量：CPU=2.0 → 归一化：2.0/4.0 = 0.5
  - 比较：0.1 < 0.5 ✓ 可满足

## 五、影响与注意事项

### 正面影响

1. **训练稳定性提升**
   - 输入特征范围统一在 `[0, 1]`
   - 减少数值溢出和梯度爆炸问题
   - 优化器收敛更稳定

2. **端到端一致性**
   - 预训练模型可直接用于fine-tuning
   - 无需特征转换或重新适配
   - 简化模型部署流程

3. **模型泛化性**
   - 不依赖具体资源值的尺度
   - 更容易适应不同规模的网络拓扑

### 注意事项

1. **旧模型不兼容** ⚠️
   - 用旧数据训练的模型无法直接使用
   - 需要用新数据集重新预训练
   - 建议删除旧的预训练checkpoint

2. **资源检查需反归一化**
   - 在 `env.py` 中已处理（见下节）
   - 放置时需将归一化需求乘以最大容量

3. **数据集版本管理**
   - 保存的数据集包含 `'normalized': True` 标志
   - 加载时检查归一化状态
   - 避免混用归一化/非归一化数据

## 六、Fine-tuning中的反归一化

在 `env.py` 中，资源检查、扣减、恢复都需要将归一化需求转回绝对值：

### `_check_node_feasible()`
```python
# 将归一化需求转为绝对值
cpu = float(feats[0].item()) * (self._sn_max_capacity['cpu_max'] + 1e-8)
mem = float(feats[1].item()) * (self._sn_max_capacity['mem_max'] + 1e-8)
disk = float(feats[2].item()) * (self._sn_max_capacity['disk_max'] + 1e-8)

# 然后与SN的绝对资源比较
if cpu > res['cpu'] + 1e-9: return False
```

### `_apply_mapping()` 和 `_release_workflow()`
同样的反归一化逻辑应用于资源扣减和恢复。
