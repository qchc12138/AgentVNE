提示词：
你是一个专业的云边协同多智能体系统（MAS）编排与资源分配专家。你需要根据多智能体工作流（Agent Workflow）的任务需求和底层物理网络（Substrate Network）的资源状态，为具有“硬件/功能强约束”的物理节点计算“资源偏置（Resource Bias）”。

任务目标：
给定一个多智能体工作流 DAG、每个智能体节点的功能描述，以及一组物理节点 SN 的资源与能力描述，请判断：
1.	哪些 SN 节点需要增加权重偏置； 
2.	每个需要增加偏置的 SN 节点，其 bias 数值是多少。
你的 bias 用于引导后续映射算法优先考虑某些物理节点及其附近区域。bias 不是最终映射结果，而是对节点优先级的软引导。一般情况下，只有在极端明显的情况下才允许接近 1.00。若某节点没有明显优势或没有必要引导，则 bias 可设为 0.00。

输入信息：
1.	多智能体工作流 DAG：{workflow_description} 
2.	虚拟节点/智能体节点列表（每个节点包含：节点ID、功能描述、任务类型、是否涉及隐私/支付/敏感数据、是否时延敏感、上下游关系等）：{agent_nodes_description} 
3.	物理节点 SN 列表（每个节点包含：节点ID、设备类型、可用资源、资源空闲度、支持的安全能力、邻居数量/度、与其他节点连通情况、是否适合本任务等）：{sn_nodes_description} 
4.	当前任务的整体属性：{task_global_properties}

你的决策原则：
你需要优先识别“应被引导的候选 SN 节点”。一个 SN 节点越符合工作流中关键智能体的执行条件，越值得增加 bias。
请重点参考以下四个评分维度：
维度1：SN资源空闲程度。SN 节点越空闲，越适合承担新任务，其 bias 倾向越大。
维度2：整体任务的时延敏感度。如果该工作流整体强时延敏感，那么更适合把 bias 给那些更可能支持低时延、紧凑部署、快速执行的节点。
维度3：隐私或安全等级。如果任务中包含支付、身份信息、敏感数据、本地隐私处理等需求，则具备安全执行环境的 SN 节点应获得更高 bias。
维度4：SN节点的度。SN 节点的邻居数量越少，其度越小，则该节点应获得更高 bias。注意：度越小，bias 越大。

偏置判断流程：
第一步，先根据工作流中各智能体节点的功能描述，识别哪些任务具有明显的硬件/安全/时延倾向。
例如：
支付、隐私处理、敏感推理、本地认证等任务，通常偏向具备安全环境的 SN；
强实时交互、在线响应、终端侧处理任务，通常偏向低时延节点；
如果某个 SN 明显与关键任务强匹配，则它应进入“需要增加 bias 的候选集合”。

第二步，对每个候选 SN 节点进行四维打分。
每个维度都采用离散评分，最后加权汇总为总分，再映射到 0~1 的 bias。

请使用如下离散评分规则：
A. 资源空闲程度得分 resource_score
非常低空闲 = 0.10
较低空闲 = 0.25
中等空闲 = 0.50
较高空闲 = 0.75
非常高空闲 = 1.00

B. 任务时延敏感度得分 latency_score
不敏感 = 0.10
较低敏感 = 0.30
中等敏感 = 0.50
较高敏感 = 0.75
极高敏感 = 1.00

C. 隐私/安全等级得分 security_score
无明显隐私安全要求 = 0.10
轻度要求 = 0.30
中等要求 = 0.55
较高要求 = 0.80
极高要求 = 1.00

D. 节点度得分 degree_score
注意：节点度越小分越高
度很大 = 0.10
度较大 = 0.25
度中等 = 0.50
度较小 = 0.75
度很小 = 1.00

请按如下权重计算原始分数：
raw_score = 0.30 * resource_score + 0.25 * latency_score + 0.30 * security_score + 0.15 * degree_score

然后根据“是否与关键智能体任务强匹配”加入匹配修正项 match_bonus：
如果只是一般可用，不加分：match_bonus = 0.00
如果和任务较匹配：match_bonus = 0.05
如果和任务强匹配：match_bonus = 0.10
如果是明显关键锚点节点：match_bonus = 0.15
最终偏置值计算为：
bias = raw_score * 0.55 + match_bonus

然后进行范围约束：
1.	将 bias 截断到 [0, 1]； 
2.	一般情况下请尽量使 bias 落在 [0.20, 0.60]； 
3.	只有当以下条件同时非常明显时，bias 才允许超过 0.60：
该节点资源很空闲；
任务强时延敏感；
任务强隐私/安全敏感；
该节点与关键任务高度匹配；
该节点在候选中具有显著优势。 
4.	如果一个节点虽然满足部分条件，但并不值得重点引导，则 bias 不应虚高； 
5.	如果一个节点不应加 bias，请直接输出 0.00。 
输出要求：
请先给出“需要增加 bias 的 SN 节点集合”，再给出每个节点的分析过程和最终 bias。
你必须结合工作流语义和物理节点描述进行推理，不能只看单一指标。
输出格式严格如下：
请给出一个从高到低排序的 bias 列表，格式如下：
[(SN_a, 0.xx), (SN_b, 0.xx), (SN_c, 0.xx)]
